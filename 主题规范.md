# misc-api 主题开发规范

本文档说明前台与用户中心主题的目录结构、**主题引入与渲染流程**、资源隔离及安全约定。管理员后台（`/admin/`）始终使用系统默认 UI，不随前台主题切换。

---

## 一、主题目录位置

所有主题位于：

```
core/theme/{主题ID}/
```

系统在启动时**自动扫描** `core/theme/` 下的一级子文件夹：若目录内存在有效的 `theme.json`，即识别为可用主题，并在后台「主题设置」中展示，**无需手动注册**。

---

## 二、标准目录结构

```
core/theme/{id}/
├── theme.json              # 主题元数据（必填）
├── preview.png             # 后台预览图（开发阶段随主题包一并部署）
├── assets/
│   ├── theme.css           # 前台页面样式（必填）
│   ├── theme.js            # 前台页面脚本（可选）
│   ├── user.css            # 用户中心完整样式（必填，含 shell + 内容区）
│   ├── user.js             # 用户中心交互（侧边栏等，可选）
│   ├── auth.css            # 认证页完整样式（必填）
│   └── auth.js             # 认证页交互（可选）
├── layout/
│   ├── header.php          # 前台顶栏 + 抽屉（本主题独立实现）
│   └── footer.php          # 前台页脚
├── pages/                  # 前台页面模板
│   ├── home.php
│   ├── apis.php
│   └── …
└── user/                   # 用户中心主题包（与前台主题联动）
    ├── layout.php          # 用户中心 shell（侧边栏 + 顶栏，本主题独立实现）
    └── auth/
        ├── layout.php      # 认证页 head/foot（仅加载本主题 auth 资源）
        ├── login.php
        ├── register.php
        ├── forgot.php
        └── bind.php
```

### 资源隔离原则（强制）

| 规则 | 说明 |
|------|------|
| **无跨主题回退** | `ThemeManager::resolveActiveThemeFile()` 仅解析**当前激活主题**内文件；缺失时显示错误提示，**不会**加载 `default` 或其他主题同名文件 |
| **无跨主题共用 shell** | 顶栏、侧边栏、抽屉、页脚等布局 HTML **必须在各主题 `layout/`、`user/layout.php` 内各自实现** |
| **样式脚本归属主题包** | 主题的 CSS、JS **必须放在该主题 `assets/` 目录**；禁止在主题中引用全局 `admin.css` / `admin.js` 作为主题 UI |
| **路由与视图分离** | 路由 PHP（如 `index.php`、`user/login.php`）只负责业务与 POST；**视图 HTML 放在主题包** |

允许引用的**全局基础资源**（非主题 UI）：

- `assets/css/common.css`、`toast.css`、`modal.css`、`icons.css`（用户中心基础组件）
- `assets/js/common.js`、`theme-picker.js`（Toast、主题切换等跨场景工具）

管理员认证页（`/admin/login`）使用全局 `auth-login.css`，**不属于**前台主题体系。

## 三、主题引入与渲染流程

### 3.1 配置与扫描

```
安装/启动
  → Config::get('frontend_theme')     # 数据库 vs_config，默认 default
  → ThemeManager::listThemes()        # glob core/theme/*/theme.json
  → ThemeManager::activeId()          # 校验 ID 格式 + theme.json 存在
```

后台保存主题：`admin/system/theme.php` → `ThemeManager::setActive($id)`  
`setActive` 校验：ID 正则 `^[a-z0-9][a-z0-9_-]{0,31}$` + `isValidTheme()`，防止注入非法路径。

### 3.2 前台页面

```
apis.php / index.php …
  → vs_frontend_page('home', '首页')
  → ThemeManager::renderBody($pageKey, $pageTitle, $pageData)
       1. buildContext() 注入 vsBase、navItems、userLoggedIn 等
       2. include core/theme/{active}/layout/header.php   # 本主题顶栏
       3. include core/theme/{active}/pages/{pageKey}.php
       4. include core/theme/{active}/layout/footer.php
       5. header 内加载本主题 theme.css / theme.js（assetUrl）
```

### 3.3 用户中心

```
user/index.php（等业务路由）
  → ThemeManager::renderUserLayoutStart($title, $activeMenu)
       → resolveActiveThemeFile('user/layout.php')   # 严格，无回退
       → vs_theme_user_layout_start()                # 本主题侧边栏 + 顶栏
       → 加载本主题 user.css / user.js
  → 路由文件输出内容区 HTML
  → ThemeManager::renderUserLayoutEnd()
```

### 3.4 认证页（登录 / 注册 / 忘记密码 / 绑定）

```
user/login.php（仅 POST 逻辑）
  → ThemeManager::renderAuthPage('login', '用户登录', $pageData)
       → resolveActiveThemeFile('user/auth/login.php')
       → 模板内调用 ThemeManager::renderThemeAuthHead($title)
            → ensureAuthLayoutLoaded() → user/auth/layout.php
            → vs_theme_auth_head()：仅本主题 auth.css + toast
       → 模板输出表单 HTML
       → ThemeManager::renderThemeAuthFoot()：common.js + 本主题 auth.js
```

### 3.5 静态资源 URL

```
ThemeManager::assetUrl($themeId, 'assets/theme.css')
  → 校验 themeId（isValidTheme）
  → 拒绝相对路径中的 ..
  → /core/theme/{id}/assets/theme.css（rawurlencode 主题 ID）
```

`resolveActiveThemeFile` 使用 `realpath()` 比对，阻止 `..` 路径穿越。

## 四、theme.json 字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 与文件夹名一致，小写字母/数字/下划线 |
| `name` | 是 | 后台展示名称 |
| `version` | 否 | 主题包版本 |
| `description` | 否 | 简短说明 |
| `preview` | 否 | 预览图相对路径，默认 `preview.png` |

示例：

```json
{
    "id": "slate",
    "name": "青绿平台",
    "version": "2.3.0",
    "description": "浅色青绿 API 平台风",
    "preview": "preview.png"
}
```

## 五、作用范围

| 区域 | 是否随 `frontend_theme` 切换 | 说明 |
|------|------------------------------|------|
| 前台页面 `/`、`/apis` … | **是** | `vs_frontend_page()` → `ThemeManager::renderBody()` |
| 用户中心 `/user/*` | **是** | `renderUserLayoutStart/End()`、`renderAuthPage()` |
| 管理员后台 `/admin/*` | **否** | 固定系统默认 UI |

切换主题后，登录、注册、忘记密码、绑定、控制台等，均加载**当前主题包**内的模板与样式。

## 六、预览图

- 路径：`core/theme/{id}/preview.png`
- 在主题开发阶段由开发者随主题包**预先部署**，供后台主题设置页展示。
- 后台读取顺序：`theme.json` 的 `preview` 字段 → 根目录 `preview.png` → 其他常见文件名。

## 七、新增主题步骤

1. 在 `core/theme/` 下新建文件夹，命名即主题 ID。
2. 编写 `theme.json`。
3. **完整实现** `assets/theme.css`、`layout/header.php`、`layout/footer.php`、`pages/*`（前台）。
4. **完整实现** `user/layout.php`、`user/auth/layout.php`、`user/auth/*.php`、`assets/user.css`、`assets/auth.css`（用户中心与认证；**不可依赖 default 回退**）。
5. 按需添加 `theme.js`、`user.js`、`auth.js`。
6. 放置 `preview.png`。
7. 后台「主题设置」保存切换，无需改数据库注册代码。

若关键文件缺失，页面将显示「主题布局缺失」类提示，便于开发期发现问题。
