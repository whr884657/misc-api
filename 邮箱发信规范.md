# misc-api · 邮箱发信规范

> **用途：** 规范验证码邮件、通知邮件的发送流程与安全策略。  
> **适用版本：** v2.10.1 起  
> **相关代码：** `core/AuthSecurity.php`、`user/forgot.php`、`user/register.php`、`admin/forgot.php`、`core/Mailer.php`

---

## 一、已发现问题（安全审计）

### 1.1 现象

在 `/user/forgot`（`action=send_code`）等验证码接口上，使用抓包工具可绕过浏览器前端倒计时，在离开页面后仍持续 POST 请求：

| 请求 | 典型返回 |
|------|----------|
| 首次 | `code: 1`，提示「如果该邮箱已注册，验证码将发送到您的邮箱…」 |
| 短时间内重复 | `code: 0`，「发送过于频繁，请 N 秒后再试」 |

攻击者可按最短间隔（旧版约 60 秒）长期轰炸同一邮箱，造成**邮箱骚扰**与 SMTP 资源浪费。

### 1.2 根因分析

| 问题 | 说明 |
|------|------|
| **仅前端倒计时** | 按钮 60s 冷却可被 API 工具直接绕过，服务端才是唯一可信限流 |
| **计数时机错误** | 忘记密码场景仅在「邮箱已注册且实际发信」后才 `recordMailCodeSent`，未注册邮箱或仅探测请求**不计入** IP/邮箱配额 |
| **间隔过短** | 同邮箱最短 60 秒，单 IP 1 小时 8 次，可持续低频轰炸 |
| **缺少 IP 级最短间隔** | 轮换多个邮箱时，仅邮箱维度间隔无法约束同一来源 |
| **缺少日限额** | 无 24 小时上限，长周期骚扰仍可能发生 |

> CSRF + 同源校验可挡跨站请求，但**无法阻止**持有合法 Session 与 Token 的脚本化重复调用。

---

## 二、解决方案（v2.10.1）

### 2.1 服务端限流策略（`AuthSecurity`）

存储：`config/.security/rate_limit.json`（运行时自动创建，勿提交 Git）

| 维度 | 常量 | 限制 |
|------|------|------|
| 同 IP 任意发信最短间隔 | `MAIL_IP_MIN_INTERVAL` | **45 秒** |
| 同邮箱最短间隔 | `MAIL_MIN_INTERVAL` | **120 秒** |
| 同 IP 1 小时 | `MAIL_IP_MAX` | **5 次** |
| 同邮箱 1 小时 | `MAIL_EMAIL_MAX` | **3 次** |
| 同邮箱 24 小时 | `MAIL_EMAIL_DAILY_MAX` | **8 次** |

**关键改动：**

1. **`recordMailCodeAttempt($email)`** — 通过 `checkMailCodeAllowed` 后**立即计入**，无论邮箱是否注册、是否真正发出邮件。
2. **检查顺序** — 先 IP 间隔 → 邮箱间隔 → 小时配额 → 日配额。
3. **统一入口** — 用户注册、用户忘记密码、管理员忘记密码均走同一套限流桶（按 IP / 邮箱聚合）。

### 2.2 业务层调用顺序

```text
校验 POST + CSRF + 同源
    → 校验邮箱格式等业务规则
    → AuthSecurity::checkMailCodeAllowed($email)   // 仅检查
    → AuthSecurity::recordMailCodeAttempt($email)  // 立即计入（防绕过）
    → 查询用户是否存在
    → 存在则 Mailer::send(...)
    → 统一成功文案（不暴露是否注册）
```

### 2.3 前端配合

- 成功发送后默认倒计时 **120 秒**（与 `MAIL_MIN_INTERVAL` 一致）。
- 限流失败时从服务端文案解析「请 N 秒后再试」，按钮同步进入倒计时，避免用户反复点击。

涉及页面：

- `core/theme/default/user/auth/forgot.php`、`register.php`
- `core/theme/slate/user/auth/forgot.php`、`register.php`
- `admin/forgot.php`

## 三、后续预防措施

### 3.1 开发规范

| 规则 | 要求 |
|------|------|
| 新增 `send_code` 类接口 | 必须调用 `checkMailCodeAllowed` + `recordMailCodeAttempt` |
| 禁止仅依赖前端倒计时 | 前端冷却仅为体验，服务端限流不可省略 |
| 忘记密码类接口 | 响应文案保持统一，**不得**因未注册返回不同提示 |
| 限流计数 | **请求通过检查后即计入**，不得仅在 `Mailer::send` 成功后才计入 |
| 调整阈值 | 修改 `AuthSecurity` 常量后，同步更新本文档与前端默认倒计时 |

### 3.2 运维建议

- 监控 `config/.security/rate_limit.json` 体积；异常膨胀时可清理过期键（保存逻辑已自动剔除 24h 外记录）。
- SMTP 侧配置发信频率与每日配额，与系统限流形成双重保护。
- 对持续触发限流的 IP，可在 Nginx / WAF 层追加封禁。
- 生产环境启用 HTTPS，保证 Session Cookie `Secure` 与 `SameSite=Lax` 生效。

### 3.3 测试检查项

- [ ] 同一邮箱 120 秒内第二次请求被拒绝，且返回剩余秒数
- [ ] 未注册邮箱连续请求仍触发 IP / 邮箱限流（旧版漏洞回归）
- [ ] 离开页面后用抓包工具 POST，无法突破服务端间隔
- [ ] 注册 / 用户忘记密码 / 管理员忘记密码共享 IP 小时配额
- [ ] 超限提示与前端按钮倒计时一致

## 四、邮箱发信运作规范

### 4.1 发信场景

| 场景 | 入口 | 邮件主题示例 |
|------|------|----------------|
| 用户注册 | `user/register.php` | `{站点名} 注册验证码` |
| 用户忘记密码 | `user/forgot.php` | `{站点名} 密码重置验证码` |
| 管理员忘记密码 | `admin/forgot.php` | `{站点名} 密码重置验证码` |

### 4.2 验证码规则

- 6 位数字，`random_int(100000, 999999)`
- Session 存储，有效期 **300 秒（5 分钟）**
- 验证码仅通过邮件下发，接口 JSON **不返回**验证码内容

### 4.3 邮件内容要求

- HTML 正文，含站点名、用途说明、验证码、有效期、非本人操作提示
- 主题简洁明确，便于用户识别
- 发信前确认后台「系统设置」已配置 SMTP（`Config::isMailEnabled()`）

### 4.4 接口安全基线

所有 `send_code` 请求须满足：

1. `POST` 方法
2. `vs_auth_require_post()` → 同源 + CSRF
3. `AuthSecurity::checkMailCodeAllowed`
4. `AuthSecurity::recordMailCodeAttempt`
5. 限流失败返回 `code: 0`，文案含可操作等待时间（秒）

### 4.5 错误文案约定

| 场景 | 文案 |
|------|------|
| IP 间隔不足 | `发送过于频繁，请 {N} 秒后再试` |
| 邮箱间隔不足 | `发送过于频繁，请 {N} 秒后再试` |
| IP 小时超限 | `发送过于频繁，请稍后再试` |
| 邮箱小时超限 | `该邮箱发送次数过多，请稍后再试` |
| 邮箱日超限 | `该邮箱今日发送次数已达上限，请明天再试` |
| 成功（忘记密码） | `如果该邮箱已注册，验证码将发送到您的邮箱，请查收（含垃圾箱）` |
